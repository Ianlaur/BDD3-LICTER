# France Zara Stores + Google Reviews Scraper — Design

**Date:** 2026-04-01
**Status:** Approved (brainstorming)
**Owner:** Zara Social Data Intelligence pipeline
**Consumers:** `dashboard.Store`, `dashboard.Review` (read by Next.js dashboard at `src/dashboard/`)

## Goal

Replace the dashboard's seed data with real Zara France data. Build the project's first end-to-end scraper: enumerate every Zara physical store in France, then collect ~50 recent French-language Google reviews per store. Output lands in the `dashboard` Postgres schema the Next.js app already reads via Prisma.

This is the J3 deliverable input ("V1 dashboard with real data") and the foundation for the J4 Loom video ("preuve d'automatisation"). N8N orchestration is already in place separately and is out of scope for this spec.

## Non-goals

- No sentiment analysis. `SentimentScore` rows are not generated by this scraper. A later agent will fill them in.
- No social platform scraping (TikTok, Instagram, Reddit, Trustpilot). Google Maps only.
- No multi-country support. France only for V1.
- No N8N workflow changes. The Python entry point exists; orchestration is wired in a separate workstream.
- No Apify. Pure Python + Selenium per the user's instruction.
- No backfill of historical reviews beyond what Google Maps surfaces in the latest 50 per store.

## Constraints

- **Stack:** Python 3.11+, Selenium with vanilla `chromedriver` (no `undetected-chromedriver`, no Playwright). User explicitly chose this trade-off, accepting slower runs in exchange for fewer dependencies and a simpler stack.
- **Volume:** ~100 stores × 50 reviews = ~5,000 reviews. French language only.
- **Politeness:** Generous delays (2–4s between actions, 8–15s between stores, 60s pause every 20 stores). Persistent Chrome `user-data-dir` so cookies survive across runs.
- **DB:** Same Neon Postgres + same `DATABASE_URL` the dashboard uses. Tables live in the `dashboard` schema (multi-schema setup is already in place).
- **Idempotency:** Re-runs must not duplicate. Enforced by `dashboard.Store.code` UNIQUE and `dashboard.Review (source, externalId)` UNIQUE constraints, plus `ON CONFLICT` clauses in the loader.

## Architecture

Three-phase pipeline. Each phase reads from the previous phase's output file (or DB) and writes to a new artifact. Each phase is independently re-runnable and resumable.

```
┌─────────────────┐    ┌──────────────────┐    ┌──────────────────┐
│  Phase 1        │    │  Phase 2         │    │  Phase 3         │
│  scrape_stores  │───▶│  scrape_reviews  │───▶│  load            │
│                 │    │                  │    │                  │
│  Zara locator   │    │  Google Maps     │    │  clean + upsert  │
│  → JSONL        │    │  → JSONL         │    │  → dashboard.*   │
└─────────────────┘    └──────────────────┘    └──────────────────┘
       │                       │                       │
       ▼                       ▼                       ▼
 stores.jsonl           reviews.jsonl           dashboard.Store
                        reviews.errors.jsonl    dashboard.Review
```

### Why three phases (not one monolithic script)

- Phase 2 is the failure-prone one. Vanilla Selenium against Google Maps will hit CAPTCHAs, layout changes, and network blips. Splitting it from phases 1 and 3 means a Phase 2 crash doesn't lose store data and doesn't roll back DB state.
- The JSONL artifacts are inspectable, diffable, and replayable. Debugging "why is this store missing reviews" is `grep store_code reviews.jsonl` instead of "re-run the whole thing and add print statements."
- Phase 3 (DB load) is fast and pure — separating it lets us re-run loads against the same JSONL when iterating on cleaning logic.

### File layout

```
src/zara/scrapers/france/
├── __init__.py
├── __main__.py              # CLI dispatcher: all | stores | reviews | load
├── scrape_stores.py         # Phase 1
├── scrape_reviews.py        # Phase 2
├── load.py                  # Phase 3
├── browser.py               # Selenium driver factory + polite-delay helpers
├── parsing.py               # DOM extractors, French date parser
└── config.py                # Paths, delays, selectors, URLs

data/raw/france/             # gitignored
├── stores.jsonl
├── reviews.jsonl
├── reviews.errors.jsonl
└── chrome-profile/          # persistent Selenium user-data-dir
```

`config.py` centralizes every magic value (selectors, delays, file paths, URLs) so adapting to a Google Maps DOM change is one file edit.

## Data model mapping

### Phase 1 output: `stores.jsonl`

One JSON object per line:

```json
{
  "code": "ZAR-FR-1234",
  "name": "Zara Paris Haussmann",
  "city": "Paris",
  "country": "FR",
  "address": "44 Boulevard Haussmann, 75009 Paris",
  "lat": 48.8738,
  "lng": 2.3326,
  "zara_locator_id": "1234",
  "scraped_at": "2026-04-01T10:23:11Z"
}
```

### Phase 2 output: `reviews.jsonl`

One JSON object per line:

```json
{
  "store_code": "ZAR-FR-1234",
  "author": "Jean D.",
  "rating": 4,
  "body": "Bon accueil mais file d'attente longue.",
  "language": "fr",
  "relative_date": "il y a 2 mois",
  "google_url": "https://www.google.com/maps/place/...",
  "scraped_at": "2026-04-01T10:48:02Z"
}
```

Note: Google Maps does not expose a stable public review ID in the DOM. We compute `externalId` deterministically in Phase 3 from the review content (see below) so the same review always hashes to the same key on re-runs.

### Phase 3: mapping to `dashboard.Store`

```
ZaraLocatorRecord  →  dashboard.Store
─────────────────────────────────────
code               →  code           (UNIQUE)
name               →  name
city               →  city
country = "FR"     →  country
address            →  address
lat, lng           →  lat, lng
(none)             →  openedAt = null
(now)              →  createdAt, updatedAt
```

Upsert via `INSERT INTO dashboard."Store" (...) ON CONFLICT (code) DO UPDATE SET ...`. Updates refresh name/address/coords in case Zara moves a store; `code` stays stable.

### Phase 3: mapping to `dashboard.Review`

```
GoogleReviewRecord  →  dashboard.Review
───────────────────────────────────────
"google"            →  source
hash(...)           →  externalId        (see below)
author              →  author
rating              →  rating
null                →  title             (Google reviews have no title)
body (cleaned)      →  body
"fr"                →  language
google_url          →  url
parsed_postedAt     →  postedAt
(now)               →  collectedAt
store.id (joined)   →  storeId
```

**externalId derivation:**

```
externalId = sha1(
  store_code || "|" ||
  author     || "|" ||
  iso_postedAt[:10] || "|" ||
  body[:120]
).hexdigest()[:32]
```

Day-precision date is intentional: Google's relative-date parser is day-accurate at best, so we don't want hour-level jitter changing the hash on re-scrape. `body[:120]` guards against authors who post the same short comment twice on different days, which is rare enough to ignore.

The compound `(source, externalId)` UNIQUE constraint is the dedup key. `ON CONFLICT (source, externalId) DO NOTHING` skips any review we already have.

## Phase details

### Phase 1 — Stores

1. Try Zara's public locator JSON endpoint with `requests` (e.g., `https://www.zara.com/itxrest/2/catalog/store/...`). If it returns 200 and the payload looks right, parse it directly — no Selenium needed for stores.
2. **Fallback:** if the endpoint shape has changed or returns an error, drive Selenium against `https://www.zara.com/fr/en/stores-locator` and scrape the rendered list.
3. Filter: keep only `country == "FR"` and physical stores (drop pickup points, lockers, popups without coordinates).
4. Map each record to the Phase 1 output schema.
5. Write atomically: write to `stores.jsonl.tmp`, then `os.rename` over `stores.jsonl`. No partial files visible to Phase 2.
6. Print a summary: `"Phase 1: 102 French stores written to data/raw/france/stores.jsonl"`.

### Phase 2 — Reviews (the hard part)

Selenium driver setup (`browser.py`):
- Vanilla `selenium.webdriver.Chrome`
- `--user-data-dir=data/raw/france/chrome-profile` (persists cookies + accepted Google consent across runs)
- Headless **off** by default (Google flags headless aggressively); a `--headless` flag is available for the Loom recording or CI but not the default
- Window size 1280×900
- `webdriver-manager` resolves the chromedriver binary

For each store in `stores.jsonl`:

1. **Skip-if-done check:** if any record with this `store_code` already exists in `reviews.jsonl`, skip. This is the resume mechanism.
2. **Search:** navigate to `https://www.google.com/maps/search/Zara+{name}+{city}` (URL-encoded).
3. **First-result click:** wait for the result list selector, click the first card.
4. **Open Reviews tab:** locate by aria-label or button text `"Avis"` / `"Reviews"`, click.
5. **Filter language:** click "Sort" → "Most relevant" (default). Language filter is enforced post-hoc in cleaning by `langdetect`; Google's UI language filter is unreliable across regions.
6. **Scroll loop:** find the scrollable reviews container, scroll to bottom, wait, count rendered review cards. Stop when `count >= 50` or three consecutive scrolls produce no new cards.
7. **Extract per card:**
   - Author display name
   - Star rating (parse from `aria-label="X stars"`)
   - "More" button click if review is truncated, then text body
   - Relative date string ("il y a 2 mois", "il y a une semaine")
   - Permalink to the review (for the `url` field)
8. **Per-store delays:** 2–4s between scrolls, 8–15s after the store completes, **60s cooldown every 20 stores**.
9. **Append:** write each store's reviews to `reviews.jsonl` (append mode, one JSON per line, flush after each store so a crash loses at most one store's worth).
10. **Errors:**
    - Store not found on Maps → write `{store_code, error: "not_found"}` to `reviews.errors.jsonl`, continue.
    - Selector not found / DOM changed → write `{store_code, error: "selector", details}` to errors file, continue.
    - CAPTCHA detected (specific URL pattern or known DOM marker) → save a screenshot, log the offending store, **exit cleanly** so the JSONL is preserved and the run can resume later.
    - Generic exception → retry once after 30s, then log to errors file and continue.

### Phase 3 — Load

1. Read `stores.jsonl`. Upsert into `dashboard.Store` via `psycopg`:
   ```sql
   INSERT INTO dashboard."Store" (id, code, name, city, country, address, lat, lng, "createdAt", "updatedAt")
   VALUES (%s, %s, %s, %s, %s, %s, %s, %s, NOW(), NOW())
   ON CONFLICT (code) DO UPDATE SET
     name = EXCLUDED.name,
     address = EXCLUDED.address,
     lat = EXCLUDED.lat,
     lng = EXCLUDED.lng,
     "updatedAt" = NOW();
   ```
   `id` is generated client-side using `cuid` (matches the Prisma schema's default).

2. Build a `code → id` lookup from the result.

3. Read `reviews.jsonl`. For each record:
   - **Normalize body:** strip whitespace, collapse multiple spaces, drop if empty after cleaning.
   - **Detect language:** run `langdetect` on the body. Drop if not French. (Belt and suspenders — the search is in France but Google still surfaces some English reviews.)
   - **Parse date:** convert "il y a 2 mois" → absolute `datetime` using `python-dateutil` + a small French regex helper. Anchor to `scraped_at` to avoid drift on re-runs.
   - **Compute externalId:** as defined above.
   - **Resolve storeId:** lookup by `store_code` in the map built in step 2.
   - **Reuse existing helpers:** `src/zara/cleaning/pipeline.py` already has `clean_review_record()` for normalization. Import and reuse it instead of duplicating logic. If its output shape doesn't match, adapt the call site in Phase 3 — do not fork the cleaner.

4. Upsert into `dashboard.Review`:
   ```sql
   INSERT INTO dashboard."Review"
     (id, source, "externalId", author, rating, body, language, url, "postedAt", "collectedAt", "storeId")
   VALUES (...)
   ON CONFLICT (source, "externalId") DO NOTHING;
   ```

5. Print a summary:
   ```
   Phase 3 complete:
     Stores upserted:    102
     Reviews inserted:   4823
     Reviews skipped:    192 (already in DB)
     Reviews dropped:    47  (empty body, non-French, or unparseable date)
   ```

## DB access

- Library: `psycopg[binary]` (Prisma's Python client is immature; raw SQL is faster to write and more reliable here).
- Connection string: `DATABASE_URL` from `src/dashboard/.env` — same Neon pooled URL the dashboard uses. Loaded via `python-dotenv` from a `.env` resolved by walking up from `src/zara/` to find the dashboard's `.env`. (Reason: there's only one Postgres in the project; we want one source of truth.)
- All SQL is fully schema-qualified: `dashboard."Store"`, `dashboard."Review"`. Quotes are required because Prisma uses PascalCase table names.
- Use a single connection per phase, autocommit off, commit at the end. Phase 3 wraps the whole load in one transaction so a partial failure rolls back cleanly.

## Error handling summary

| Failure mode               | Behavior                                          | Recovery                                |
| -------------------------- | ------------------------------------------------- | --------------------------------------- |
| Network blip mid-action    | Retry once after 30s backoff, then skip + log    | Re-run phase 2 → resume from JSONL      |
| Store not found on Maps    | Log to `reviews.errors.jsonl`, continue          | Manual fixup, or accept missing store   |
| DOM selector missing       | Log to `reviews.errors.jsonl`, continue          | Update `config.py` selectors, re-run    |
| CAPTCHA detected           | Screenshot, clean exit, JSONL preserved          | Wait, re-run phase 2 (resume)           |
| Phase 1 endpoint changes   | Fallback to Selenium against the locator page    | Locator page still works                |
| Phase 3 DB constraint hit  | Transaction rolls back, error printed            | Inspect, fix data or schema, re-run     |
| Chromedriver version drift | `webdriver-manager` auto-updates                 | None needed                             |

## Testing

This is hackathon scope, so the testing strategy is pragmatic, not exhaustive:

- **Phase 1 smoke test:** `--dry-run` flag prints store count + first 3 records without writing the JSONL. Run before the real scrape to confirm the Zara endpoint still works.
- **Phase 2 smoke test:** `--limit 5` flag scrapes only the first 5 stores. Use this to verify selectors before committing to a full run. ~5 minutes.
- **Phase 3 smoke test:** `--dry-run` flag executes all parsing/cleaning and prints the planned `INSERT` counts but rolls back instead of committing. Run before every load.
- **Manual verification after a real run:**
  ```sql
  SELECT COUNT(*) FROM dashboard."Store"  WHERE country = 'FR';
  SELECT COUNT(*) FROM dashboard."Review" WHERE source  = 'google';
  SELECT name, COUNT(r.id) AS reviews
  FROM dashboard."Store" s LEFT JOIN dashboard."Review" r ON r."storeId" = s.id
  WHERE s.country = 'FR' GROUP BY name ORDER BY reviews DESC LIMIT 10;
  ```
- **No unit tests** for selector logic — they go stale the moment Google ships a UI tweak. Selectors live in `config.py` so they're one edit away.

## Open questions

None at design time. Selector strings will be filled in during implementation by running Selenium interactively against a real Google Maps page.

## What this enables next

Once this scraper is in place and has populated `dashboard.Store` + `dashboard.Review` with real French data:

1. **Sentiment agent** can run a separate pass over `dashboard.Review WHERE NOT EXISTS (SELECT 1 FROM dashboard."SentimentScore" WHERE reviewId = ...)` and fill in `SentimentScore`.
2. **Trend agent** can aggregate topics from sentiment output into `dashboard.Trend`.
3. **Crisis-detection agent** can compare rolling sentiment windows and emit `dashboard.Alert` rows.
4. The dashboard's seed data can be deleted in favor of real data (separate ticket).
5. Same scraper structure can be cloned for other countries (`src/zara/scrapers/spain/`, `germany/`, …) by changing the `country` filter in Phase 1 and the search URL in Phase 2.
